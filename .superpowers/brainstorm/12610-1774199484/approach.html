How should we build Math or Die?

Three approaches — same end result, different trade-offs

A

Vanilla JS + CSS

Pure HTML, CSS, and JavaScript. No build step — just open the file in Safari on iPad. All game state managed manually with JS objects and DOM manipulation.

Pros

  • No setup, no dependencies
  • Open directly in Safari — no server needed
  • Easy to hand off or tweak later

Cons

  • Complex game state gets messy fast
  • Harder to manage timer + enemy AI + UI updates simultaneously
B

React + Vite ⭐ Recommended

Component-based architecture. Each game element (stat bar, numpad, timer, button) is its own component. React hooks manage timers, enemy AI, and game state cleanly.

Pros

  • Clean state management — perfect for a game with many moving parts
  • Components map 1:1 to the Paper design layers
  • Easy to add features later (progression map, upgrades)
  • Runs perfectly in Safari via local dev server or static hosting

Cons

  • Requires Node.js / npm to build
  • Slightly more setup than vanilla
C

Phaser.js

A game-specific JavaScript framework with built-in scene management, tweens, timers, and sprite handling. Renders to a Canvas element.

Pros

  • Built for games — timers, animations, audio all built in
  • Scene system great for Front Page → Game Page flow

Cons

  • Canvas-based — harder to match the pixel-perfect Paper design
  • Heavier framework, more to learn
  • DOM-style UI (buttons, numpad) is awkward in Phaser
